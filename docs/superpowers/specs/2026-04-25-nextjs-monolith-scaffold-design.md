# Next.js Monolith Scaffold Design

## Context

The repository is currently documentation-first and does not yet contain the application scaffold. The product spec and implementation plan both establish that the application should begin as a single server-side Next.js App Router monolith written in TypeScript, with Docker Compose and supporting infrastructure added as the platform grows.

The immediate goal of this work is narrower than the full platform bootstrap: create the initial Next.js application scaffold in an isolated `.worktrees/` worktree using the command-line scaffold tool, verify that it runs, and merge that clean scaffold back to `main`.

## Goal

Create the first working application scaffold for Intellectual Kinetic as a TypeScript Next.js App Router project that:

1. is created with the standard Next.js CLI tooling rather than by hand
2. lives temporarily in a `.worktrees/` worktree during setup
3. boots successfully with the generated scripts
4. is ready for the later Docker Compose and persistence work defined in the product implementation plan

## Non-Goals

This step does not yet implement:

- Docker Compose
- Postgres or Redis
- persistence or project storage
- Tiptap, OCR, ingestion, or export workflows
- design-system implementation beyond the default scaffold

## Options Considered

### Option 1: Recommended - standard `create-next-app` scaffold in `.worktrees/`

Use the official `create-next-app` CLI to generate a TypeScript App Router project in a dedicated `.worktrees/` worktree, keep the scaffold minimal, verify it starts cleanly, then merge it back to `main`.

**Why this is recommended:** it is the fastest and lowest-risk path, preserves standard Next.js conventions, and matches the product spec's monolith-first direction without prematurely adding unrelated infrastructure decisions.

### Option 2: more opinionated scaffold now

Generate the app and immediately add extra foundations such as Tailwind customization, Docker Compose, and initial library structure.

**Trade-off:** it accelerates later work, but it expands the current task and increases the chance of mixing scaffold concerns with architecture work that should stay in later implementation steps.

### Option 3: manual scaffold

Create `package.json`, Next.js configuration, TypeScript configuration, and the app directory manually without the official scaffold tool.

**Trade-off:** this gives tighter control over generated files, but it is slower, easier to get wrong, and directly conflicts with the user's requirement to use the proper command-line scaffolding tool.

## Chosen Design

### Workspace strategy

Create a dedicated git worktree under `.worktrees/` for the scaffold work. The scaffold is created there first so the repository root stays stable until the generated app is verified and ready to merge back into `main`.

### Scaffold shape

Use the official Next.js CLI to create:

- Next.js App Router
- TypeScript
- server-side monolith baseline
- standard linting generated by the scaffold
- a minimal `src/`-based layout if supported by the scaffold options

The generated app should remain intentionally minimal so it becomes the clean foundation for the later Compose, persistence, ingest, review, and export tasks already defined in the product implementation plan.

### Initial repository outcome

After merging back to `main`, the repository should contain a working Next.js application scaffold with the standard generated files and scripts, plus only the minimal cleanup needed to keep the scaffold coherent and bootable in this codebase.

## Verification

The scaffold is considered complete for this step when:

1. the app is created successfully by the CLI in the `.worktrees/` worktree
2. dependencies install successfully
3. the generated validation commands provided by the scaffold complete successfully
4. the app boots successfully with the standard Next.js dev or production-start path
5. the worktree branch is merged back to `main`

## Follow-On Work

Once this scaffold lands, the next implementation step remains the existing product-plan Task 1 expansion work:

- add Docker Compose
- add Postgres and Redis containers
- define persistent volumes
- begin shaping the repository for the later persistence and review workflow features
